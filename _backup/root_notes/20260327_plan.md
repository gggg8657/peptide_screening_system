| 모듈 | 현재 | 변경 |


### 1.0 사전 수정 (Blocking Issues)

수정 없이는 파이프라인이 동작하지 않는 critical 이슈.

#### 1.0.1 LocalModelRunner 환경변수 누락

**문제**: `local_runner.py`의 `_build_env()`가 `CUDA_VISIBLE_DEVICES`만 설정. RFdiffusion 등 DGL 기반 모델에 `DGLBACKEND=pytorch`가 필요하나 전달 안 됨.

**수정 대상**: `pipeline_local/core/local_runner.py` — `_build_env()`

**수정 내용**:
```python
env["DGLBACKEND"] = "pytorch"
env["TOKENIZERS_PARALLELISM"] = "false"
env["OMP_NUM_THREADS"] = "4"
```

**검증**: RFdiffusion wrapper가 파이프라인 내에서 정상 호출되는지 확인.

#### 1.0.2 subprocess.run → Popen 교체 ⚡ [CODE 리뷰 추가]

**문제**: `subprocess.run(..., timeout=timeout)` 사용 시 `TimeoutExpired` 발생하면 `conda run` 자식 프로세스가 SIGKILL되지 않을 수 있음. GPU 메모리 누수 위험.

**수정 대상**: `pipeline_local/core/local_runner.py` — `_execute()`

**수정 내용**:
```python
proc = subprocess.Popen(cmd, stdout=PIPE, stderr=PIPE, text=True, env=env)
try:
    stdout, stderr = proc.communicate(timeout=timeout)
except subprocess.TimeoutExpired:
    proc.kill()
    proc.wait()  # zombie 방지
    raise TimeoutError(...)
```

**예상 소요**: 30분

#### 1.0.3 Step04 ESMFold 배치 모드

**문제**: 128개 서열을 한 번에 하나씩 subprocess 호출 → 각 호출마다 모델 재로드 (~10초 × 128 = ~20분).

**해결 방안**: wrapper_scripts/run_esmfold.py에 **배치 모드** 추가.
- `--input-json`에 sequences 배열 전달
- 모델 1회 로드 후 전체 서열 예측
- GPU 메모리 충분 (96GB, ESMFold은 ~4GB)

**수정 대상**: `wrapper_scripts/run_esmfold.py`, `steps/step04_qc.py`

**예상 소요**: 1~1.5시간 [CODE 리뷰 재평가]

#### 1.0.4 RFdiffusion 파이프라인 호출 검증

**문제**: 직접 호출은 성공(GPU 2번, 31KB PDB 생성)하지만 step02에서 호출 시 실패.
- 원인 1: DGLBACKEND 환경변수 미전달 (1.0.1에서 해결)
- 원인 2: max_workers=1로 변경했으나 검증 안 됨

**검증 방법**: 1.0.1 수정 후 step02 단독 실행 테스트.

#### 1.0.5 절대경로 하드코딩 환경변수화 ⚡ [CODE 리뷰 추가]

**문제**: `orchestrator.py:47-50`에 절대경로 하드코딩:

**수정 대상**: `pipeline_local/orchestrator.py`, `pipeline_local/backend/state.py`

**수정 내용**: 환경변수 `AG_SRC_REPO` 또는 config 파일에서 읽도록 변경. 현재 경로를 기본값으로 유지.

**예상 소요**: 15분

#### 1.0.6 ESMFold 이황화결합 미처리 ⚡ [SCIENCE 리뷰 추가]

**문제**: ESMFold는 Cys3-Cys14 이황화결합을 형성하지 않음. SST-14는 고리형 펩타이드인데 선형으로 구조 예측됨. 이로 인해:
- SG-SG 거리가 2.5Å 초과 → disulfide gate 전원 실패
- pLDDT도 선형 구조 기준이라 낮게 측정

**해결 방안 (2단계)**:
1. **즉시**: Step04 QC에서 disulfide gate를 **ESMFold 후 PyRosetta 이황화결합 형성** 후 적용하도록 순서 변경
2. **장기**: Step04와 Step06 사이에 PyRosetta `CyclizeMover` + `DisulfideInsertionMover` 추가

**수정 대상**: `pipeline_local/steps/step04_qc.py`

**예상 소요**: 1시간

#### 1.0.7 pLDDT 기준값 불일치 ⚡ [SCIENCE 리뷰 추가]

**문제**: 계획 본문에 "pLDDT ≥75"라 명시했으나, 실제 config `gate_thresholds.yaml`에서 `esmfold_plddt_min: 50`일 수 있음. 또한 wrapper의 pLDDT 스케일 수정(0~1 → 0~100)이 반영됐지만 검증 안 됨.

**수정 대상**: `pipeline_local/config/gate_thresholds.yaml` 확인 및 통일

### 1.7 LLM 모델을 통한 피드백 루프 확인

#### Agentic AI System 관점

현재 5개 에이전트의 LLM 활용도:

| 에이전트 | LLM 역할 | 폴백 모드 |
|---------|---------|----------|
| **PlannerAgent** | 가설 생성, 파라미터 계획 | 규칙 기반 기본값 |
| **ScientistCriticAgent** | 실패 분석, 파라미터 제안 | FAILURE_ACTION_MAP |
| **ReporterAgent** | Markdown 보고서 생성 | 템플릿 기반 |
| **QCRankerAgent** | 랭킹 최적화 (선택적) | 가중합 점수 |
| **DiversityManagerAgent** | 다양성 평가 (선택적) | 서열 기반 중복제거 |

#### 검증 항목
1. **LLM 호출 성공**: Ollama(qwen3:8b) 응답이 올바른 JSON 형식인지
2. **피드백 반영**: Critic의 파라미터 변경 제안 → Planner가 다음 iteration에 반영
3. **수렴 판단**: ddG 개선 없이 patience 초과 시 조기 종료
4. **폴백 동작**: LLM 실패 시 규칙 기반 폴백이 정상 동작하는지
5. **에이전트 성공률 추적**: `AgentResponse.used_stub` 필드 추가 [CODE 리뷰]

#### AI Scientist 관점
- Planner의 가설이 과학적으로 타당한지 (SSTR2 결합 포켓 특성 반영)
- Critic의 실패 분석이 정확한지 (pLDDT 낮음 → 서열 안정성 문제 판단)
- Reporter의 보고서가 실험 재현에 충분한 정보를 담는지
- **D-아미노산 치환 탐색이 누락됨** — DOTATATE의 D-Phe, D-Trp이 SSTR2 선택성 핵심 [SCIENCE 리뷰]

#### 2.0.1 `run_single_iteration()` 분해

현재 350줄 단일 메서드 → `_phase_*` 메서드로 분리:
```python
def run_single_iteration(self, iteration, previous_results):
    ctx = self._phase_planner(iteration, previous_results)
    ctx = self._phase_builder(ctx)       # Step01~07
    ctx = self._phase_qc_ranker(ctx)
    ctx = self._phase_diversity(ctx)
    ctx = self._phase_critic(ctx)
    ctx = self._phase_reporter(ctx)
    return ctx
```
**이유**: 2.2 Agent flow config가 이 메서드의 순서를 동적으로 변경해야 하므로 분해가 전제조건.

**예상 소요**: 2시간 [CODE 리뷰 재평가]

#### 2.0.2 `_BoundOllamaProvider` 래퍼 작성 [CODE 리뷰 추가]

**문제**: `OllamaProvider.__init__`는 `(model, base_url, timeout)`만 받고 `temperature`/`max_tokens`는 `generate()` per-call 인자. per-agent 설정을 생성자에 바인딩하려면 래퍼 필요.

**원본 AG_src 불변 원칙**을 지키면서 해결:
```python
# pipeline_local/llm_factory.py
class _BoundOllamaProvider(OllamaProvider):
    def __init__(self, *, model, base_url, timeout, temperature, max_tokens):
        super().__init__(model=model, base_url=base_url, timeout=timeout)
        self._default_temp = temperature
        self._default_max_tokens = max_tokens
    def generate(self, prompt, *, temperature=None, max_tokens=None, **kw):
        return super().generate(prompt,
            temperature=temperature or self._default_temp,
            max_tokens=max_tokens or self._default_max_tokens, **kw)
```

**예상 소요**: 30분
