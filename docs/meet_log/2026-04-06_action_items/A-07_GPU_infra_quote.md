## 배경

회의록 §2.4에서 **디퓨전 모델 기반 도킹 가속화** 방안이 논의되었다.  
Rosetta 대비 약 10배 빠른 처리 속도가 기대되나, 단일 모델 로딩 기준 **VRAM 120GB 이상**이
필요하다는 제약이 있다.

현재 프로젝트 GPU 환경은 아래와 같다:

| 구분 | 사양 | 비고 |
|------|------|------|
| 사용자 로컬 | H100 NVL × 4 (각 96GB, 총 384GB) | `CUDA_VISIBLE_DEVICES=2` 기본 설정 (`~/.zshrc`) |
| 외부망 서버 | H100 × 8 (80GB × 8, 총 640GB) | 회의록 §2.3 — 전체 시스템 배포 완료 |

외부망 H100 ×8 서버가 단일 노드 NVLink로 통합 가능하다면 640GB 총 VRAM 확보가 가능하므로,  
**추가 구매 전 기존 인프라 활용 가능 여부를 선결**해야 한다.

## 수행 방법 (단계별)

### Step 1. 기존 H100 ×8 서버 VRAM 점검

```bash
# 외부망 서버 접속 후 실행
nvidia-smi --query-gpu=index,name,memory.total,memory.free --format=csv
nvidia-smi topo -m   # NVLink topology 확인
```

확인 항목:
- GPU별 free VRAM (다른 작업 점유 여부)
- NVLink 연결 토폴로지 (단일 fabric 통합 가능 여부)
- 현재 실행 중인 프로세스 (`nvidia-smi pmon`)

### Step 2. 디퓨전 모델 PoC VRAM 실측

> **선행 조건**: A-06 (디퓨전 모델 도킹 PoC) 실행 결과

```bash
# DiffDock-PP 또는 대상 디퓨전 모델 VRAM 측정 예시
watch -n 1 nvidia-smi --query-gpu=memory.used,memory.free --format=csv,noheader
```

기록 항목:
- 모델 로딩 시 peak VRAM
- 배치 추론 시 peak VRAM
- 단일 GPU 충분 여부 vs. multi-GPU 필요 여부

### Step 3. 견적 수집 (사용자 서호성/안기범 담당)

최소 **2개 벤더**로부터 아래 옵션의 공식 견적 수집:

- **옵션 A**: NVIDIA DGX H100 (공식 NVIDIA 또는 공인 리셀러)
- **옵션 B**: NVIDIA DGX B200 (가용 여부 확인)
- **옵션 C**: 자체 빌드 H100 서버 (OEM 벤더: HPE, Dell, Supermicro 등)

### Step 4. 비교 매트릭스 완성 및 의사결정

아래 템플릿에 수집된 데이터 기입 후 팀 공유.

## 판단 기준

```
기존 H100 ×8 NVLink 통합 + A-06 PoC VRAM ≤ 640GB
    → 추가 구매 보류, 기존 인프라 활용

A-06 PoC VRAM > 640GB 또는 NVLink 통합 불가
    → DGX 구매 vs 자체 빌드 ROI 비교
        DGX H100: 즉시 검증된 솔루션, 높은 초기 비용
        DGX B200: 단일 GPU VRAM 180GB (단일 모델 로딩 최적), 더 높은 비용
        자체 빌드: 유연성, 납기 리스크
```

├── H100 × 8 (80GB × 8, 총 640GB)
└── NVLink 통합 시 단일 메모리 공간으로 활용 가능 여부 확인 필요
