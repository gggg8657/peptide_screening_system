## 1. 실험 목표

### 1.1 배경

AG_src는 인간 소마토스타틴 수용체 2형(SSTR2)을 표적으로 하는 신규 펩타이드 바인더 발굴을 위해 설계된 에이전틱 AI 시스템이다. 이 시스템은 6개의 전문 에이전트가 12단계 컴퓨팅 파이프라인(수용체 준비, 백본 생성, 서열 설계, ESMFold 품질 검증, 도킹, Rosetta 정제, 선택성 스크리닝, 분석/시각화)을 조율하며, 수렴 감지 기반의 반복 개선 주기를 수행한다.

핵심 오케스트레이터(`run_pipeline_live.py`)가 모든 에이전트와 단계를 조율하며, 각 에이전트(`AG_src/agents/*.py`)는 `BaseAgent` 추상 인터페이스의 `execute()` 메서드를 구현한다.

### 1.2 검증 목표

본 검증 실험은 세 가지 상호 관련 질문에 답한다:

1. **아키텍처 타당성**: 현재 6개 에이전트 분해(Planner, Builder, QC&Ranker, DiversityManager, Critic, Reporter)가 SSTR2 바인더 설계 작업에 최적인가, 아니면 통합된 4개 에이전트 하이브리드가 비용-성능 측면에서 더 우수한가?

2. **LLM 적합성**: 온프레미스 배포 제약(7B~11B 파라미터 모델) 하에서 어떤 에이전트가 실제로 LLM 추론을 필요로 하고, 어떤 에이전트가 결정론적 코드 모듈로 충분한가?

3. **최적 파이프라인 아키텍처**: (1)과 (2)의 결과를 종합하여, 반복당 LLM 호출을 최소화하면서 과학적 엄밀성을 유지하는 구체적 아키텍처 권고안을 도출한다.

## 2. 방법론

### 2.1 경쟁 가설 패턴

Claude Code 에이전트 팀을 활용한 **경쟁 가설** 패턴을 적용하였다. 4명의 전문 평가자가 AG_src 코드베이스를 독립 분석한 후 교차 검증을 수행하여, 적대적 엄밀성을 확보하고 확증 편향을 줄였다.

### 2.2 에이전트 팀 구성

| 에이전트 | 역할 | 임무 |
|---------|------|------|
| **advocate-6agent** | 6개 에이전트 옹호자 | 현재 6개 에이전트 분해의 유지를 주장. 관심사 분리(SoC), 테스트 용이성, 확장성 패턴 식별 |
| **advocate-hybrid** | 4개 에이전트 하이브리드 옹호자 | 4개 에이전트(Planner, MD Agent, Critic, Reporter)로의 통합을 주장. 중복성, LLM 비용 낭비, 과잉 설계 식별 |
| **llm-evaluator** | LLM 적합성 분석가 | 각 에이전트의 인지 부하를 8B~11B 모델 능력과 대비 평가. 에이전트별 LLM 필요성 점수 산출 |
| **devil-advocate** | 악마의 대변인 / 종합자 | 양쪽 아키텍처에 도전. 숨겨진 위험 식별, 하이브리드 대안 제안, 최종 권고안 종합 |

### 2.3 평가 프로토콜

각 에이전트가 독립적으로 수행한 작업:

1. `AG_src/agents/`, `AG_src/pipeline/`, `AG_src/schemas/` 전체 소스 파일 정적 분석
2. 오케스트레이터의 반복 실행 흐름 데이터 플로우 추적
3. 각 에이전트의 `execute()` 구현을 `_invoke_agent()` 스텁과 대비 평가
4. `BaseAgent` 인터페이스(`base_agent.py`)와 실제 에이전트 호출의 교차 참조
5. 독립 발견사항 산출 후 종합 라운드 수행

## 3. 아키텍처 비교 분석

### 3.1 방안 A: 6개 에이전트 아키텍처 (기존 구현)

#### 3.1.1 에이전트 목록

| 에이전트 | 모듈 | 줄 수 | 핵심 책임 |
|---------|------|-------|----------|
| **Planner** | `agents/planner.py` | 428 | 실험 계획 생성/갱신, 가설 생성, 파라미터 관리 |
| **Builder** | `agents/builder.py` | 460 | Step01~07 실행 조율, 재시도/폴백 정책, 실행 로깅 |
| **QC&Ranker** | `agents/qc_ranker.py` | 490 | 4단계 게이트 적용(pLDDT, 도킹, Rosetta, 선택성), 가중 점수화, CSV 내보내기 |
| **DiversityMgr** | `agents/diversity_manager.py` | 566 | 구조 클러스터링(FoldMason/서열), 다양성 세트 선택, 중복 감지 |
| **Critic** | `agents/critic.py` | 528 | 실패 분류(6개 유형), FAILURE_ACTION_MAP 기반 파라미터 변경 제안(최대 2개), 가설 생성 |
| **Reporter** | `agents/reporter.py` | 624 | PyMOL 4패널 렌더 스크립트, 마크다운 요약/실험 노트, 순위 테이블 CSV |

#### 3.1.2 장점

**장점1. 관심사 분리(SoC).** 각 에이전트가 단일하고 명확한 책임을 캡슐화한다. `BaseAgent` 추상 클래스(194줄)가 균일한 인터페이스(`execute()`, `send_message()`, `receive_message()`)를 강제하여 에이전트 로직과 조율 로직을 분리한다. 깔끔한 데이터 흐름이 이를 증명한다:

```
Planner → Builder → QC&Ranker → DiversityManager → Critic → Reporter
```

**장점2. 독립 테스트 가능성.** 각 에이전트의 `execute()` 메서드가 컨텍스트 딕셔너리를 받고 결과 딕셔너리를 반환하므로, 전체 파이프라인을 인스턴스화하지 않고도 개별 단위 테스트가 가능하다.

**장점3. 개방-폐쇄 원칙(OCP) 확장성.** 새 에이전트(예: 독성 스크리너) 추가 시 새 `BaseAgent` 하위 클래스 생성과 해당 분기 추가만으로 충분하며, 기존 에이전트 수정이 불필요하다.

**장점4. Critic-Planner 분리.** Critic(실패에 대한 분석적 추론)과 Planner(파라미터 조정 실행)의 아키텍처적 분리가 과학적 방법론을 반영한다. Critic의 `FAILURE_ACTION_MAP`(6개 실패 유형, 13개 교정 조치)은 통합 에이전트에서는 상실될 체계적 추론 발판을 제공한다.

**장점5. DiversityManager의 독립적 품질 게이트 역할.** DiversityManager는 QC 통과 후보에 대해 알고리즘 클러스터링을 수행한다. 분리를 통해 다양성 보장이 QC&Ranker의 점수 함수에 편향되지 않는다.

#### 3.1.3 단점

**단점1. 오케스트레이터 복잡도.** 오케스트레이터가 Builder 에이전트가 처리해야 할 조율 로직을 일부 중복으로 담고 있어 "신(God) 오케스트레이터" 패턴이 발생한다.

**단점2. 반복당 6회 LLM 호출.** 6개 에이전트 모두 LLM 지원 시, 반복당 6회 LLM 왕복이 필요하다. 일반적인 8B 모델 추론 지연(단일 A100에서 호출당 2~5초) 기준으로 반복당 순수 LLM 오버헤드가 12~30초 추가된다.

**단점3. `_invoke_agent()` 스텁 간극.** 기존 `_invoke_agent()` 구현이 규칙 기반 스텁으로, 실제 `BaseAgent.execute()` 구현과 연결되지 않았다. 즉, 정교하게 설계된 에이전트 클래스(`PlannerAgent`, `ScientistCriticAgent` 등)가 **실제로는 사용되지 않았다**. (이 문제는 이후 해결 완료)

**단점4. 에이전트 간 메시징 미사용.** `BaseAgent.send_message()` / `receive_message()` 인프라가 완전 구현되었으나 오케스트레이터에서 한 번도 호출되지 않았다.

### 3.2 방안 B: 4개 에이전트 하이브리드

#### 3.2.1 제안된 통합

| 에이전트 | 통합 원본 | LLM 크기 | 책임 |
|---------|----------|---------|------|
| **Planner** | Planner | 8B | 실험 계획, 가설, 파라미터 갱신 |
| **MD Agent** | Builder + QC&Ranker + DiversityMgr | 11B | 단계 실행, 게이트 적용, 순위화, 다양성 필터링 |
| **Critic** | Critic | 11B | 실패 분석, 파라미터 변경 제안 |
| **Reporter** | Reporter | 8B | 시각화 스크립트, 마크다운 보고서 |

#### 3.2.2 장점

**장점1. md_agent 설계 의도 정렬.** 원래 시스템의 개념 설계가 "분자 동역학 에이전트"를 구상했으며, Builder + QC&Ranker + DiversityMgr 통합으로 이 의도를 복원한다.

**장점2. LLM 호출 33% 감소.** 반복당 6회에서 4회로 감소하여 4~10초의 추론 오버헤드를 절약한다.

**장점3. 오류 복구 통합.** 현재 오류 처리가 오케스트레이터의 재시도 로직과 Builder의 `handle_failure()` 메서드에 분산되어 있는데, 통합 MD Agent가 실행과 복구를 단일 범위에서 담당한다.

#### 3.2.3 단점

**단점1. MD Agent 과부하.** 통합된 에이전트가 1,516줄의 로직(Builder 460 + QC&Ranker 490 + DiversityMgr 566)을 결합하여, 800줄 가이드라인을 위반하고 너무 많은 실패 모드를 집중시킨다.

**단점2. 자기 QC 편향.** 파이프라인을 실행하는 에이전트가 품질까지 평가하면 내재적 이해 충돌이 발생한다. 분리된 QC&Ranker는 이 편향을 구조적으로 제거한다.

**단점3. 장애 영향 범위 확대.** 다양성 클러스터링 로직의 버그가 단계 실행과 품질 게이팅까지 중단시킨다. 6개 에이전트 아키텍처에서는 이러한 버그가 `diversity_manager.py`에 격리된다.

**단점4. 11B 모델 착시.** 4절에서 상세히 다루듯, Llama 3.2 11B Vision 모델은 순수 텍스트 벤치마크에서 8B 모델 대비 무의미한 성능 향상만 제공한다.

### 3.3 방안 C: 6개 에이전트 + 하이브리드 (LLM 3개 + 코드 3개) ← **채택**

6개 에이전트 구조적 분해를 유지하되, **에이전트의 절반이 LLM 추론 없이도 충분한 순수 알고리즘 작업**을 수행한다는 점을 인식한 방안이다.

| 에이전트 | 실행 방식 | 모델/엔진 | 근거 |
|---------|----------|----------|------|
| **Planner** | LLM | Qwen 2.5 7B | 자연어 가설 생성, 창의적 파라미터 탐색, Critic 피드백 해석 필요 |
| **Builder** | 코드 (결정론적) | 규칙 기반 | 단계 시퀀싱, 재시도/폴백, 실행 로깅이 ExperimentPlan 기반으로 완전 결정론적 |
| **QC&Ranker** | 코드 (결정론적) | 순수 알고리즘 | 게이트 임계값 = 수치 비교, 가중 점수화 = 선형 대수. 추론 불필요 |
| **DiversityMgr** | 코드 (결정론적) | 순수 알고리즘 | 탐욕적 클러스터링, 쌍별 서열 동일성, 대표 선택이 표준 알고리즘 |
| **Critic** | LLM | Qwen 2.5 7B | 실패 모드에 대한 인과 추론, 창의적 가설 생성, 구조적 해석 필요 |
| **Reporter** | LLM | Qwen 2.5 7B | 실험 노트북 자연어 종합, 맥락 의존적 시각화 파라미터 선택 필요 |

#### 3.3.1 반복당 LLM 호출 예산

```
반복 N:
  [1] Planner.execute()       → Qwen 2.5 7B   (계획 + 가설)
  [2] Builder.execute()       → 코드           (Step01~07)
  [3] QCRanker.execute()      → 코드           (게이트 + 순위화)
  [4] DiversityMgr.execute()  → 코드           (클러스터링)
  [5] Critic.execute()        → Qwen 2.5 7B   (분석 + 제안)
  [6] Reporter.execute()      → Qwen 2.5 7B   (보고서 생성)

총 LLM 호출: 반복당 3회 (방안 A 대비 50% 감소)
총 코드 호출: 반복당 3회 (지연 오버헤드 제로)
```

#### 3.3.2 방안 A, B 대비 장점

| 기준 | 방안 A (6-LLM) | 방안 B (4개 에이전트) | 방안 C (6개 에이전트 하이브리드) |
|------|---------------|------------------|-----------------------------|
| 반복당 LLM 호출 | 6 | 4 | **3** |
| 에이전트 수 | 6 | 4 | **6** |
| 관심사 분리 유지 | 예 | 부분적 | **예** |
| QC 독립성 | 예 | 아니오 (자기 QC 편향) | **예** |
| 최대 모듈 크기 (줄) | 566 | ~1,516 | **566** |
| 장애 영향 범위 | 소 | 대 | **소** |
| 테스트 가능성 | 높음 | 중간 | **높음** |
| 확장성 (OCP) | 높음 | 중간 | **높음** |
| 반복당 추론 비용 | 높음 | 중간 | **낮음** |

## 4. LLM 적합성 매트릭스

### 4.1 후보 모델 벤치마크

온프레미스 배포 호환성(단일 A100 40GB 또는 이중 A6000 48GB 구성) 기준으로 다음 모델들을 평가하였다:

| 모델 | 파라미터 | MMLU | HumanEval | 컨텍스트 | 도구 호출 | JSON 구조 출력 | 추론 속도 (A100) |
|------|---------|------|-----------|---------|----------|-------------|----------------|
| Llama 3.1 8B-Instruct | 8B | 69.4 | 72.6 | 128K | 기본 | 중 (프롬프팅 필요) | ~40 tok/s |
| Llama 3.2 11B Vision-Instruct | 11B | ~69~73 | 8B 수준 | 128K | 기본 | 중 | ~30 tok/s |
| Gemma 2 9B | 9B | 71.3 | 40.2 | 8K | 제한적 | 중 | ~35 tok/s |
| **Qwen 2.5 7B-Instruct** | 7B | **74.2** | 57.9 | **128K** | **네이티브** | **강 (JSON 모드)** | ~45 tok/s |
| Mistral 7B v0.3-Instruct | 7B | 63.5 | 보통 | 32K | 네이티브 | 중 | ~45 tok/s |

#### 핵심 발견: Llama 3.2 11B Vision의 텍스트 성능 동등성

Llama 3.2 11B Vision-Instruct 모델은 추가 3B 파라미터를 주로 비전 인코더(ViT-H)와 이미지 이해용 교차 어텐션 레이어에 할당한다. 순수 텍스트 벤치마크(MMLU, HumanEval, GSM8K)에서는 Llama 3.1 8B-Instruct와 오차 범위 내의 성능을 보인다. AG_src 파이프라인은 이미지 입력을 처리하지 않으므로(PDB 파일, FASTA 서열, 수치 점수가 모두 텍스트/구조화 데이터), **11B 모델은 이 작업에서 측정 가능한 이점을 제공하지 않는다.**

#### 핵심 발견: Qwen 2.5 7B가 구조화된 에이전트 작업에 최적

Qwen 2.5 7B-Instruct가 7~8B 클래스에서 선두인 항목:
- **MMLU** (74.2): 가장 강한 범용 추론 능력
- **네이티브 도구 호출**: 내장 `<tool_call>` 형식으로 프롬프트 엔지니어링 부담 경감
- **JSON 구조 출력**: 에이전트 컨텍스트 딕셔너리에 중요한 안정적 JSON 모드 생성
- **128K 컨텍스트 윈도우**: 다중 반복 상태 축적에 충분

### 4.2 에이전트별 LLM 적합성 평가

각 에이전트를 5개 차원으로 평가하였다 (0~5점, 5 = LLM 능력 필수):

| 에이전트 | 자연어 생성 | 인과 추론 | 창의적 탐색 | 구조 출력 파싱 | 수치 연산 | **LLM 필요 점수** | **판정** |
|---------|-----------|----------|-----------|-------------|---------|----------------|---------|
| Planner | 4 | 3 | 4 | 3 | 1 | **3.0** | LLM 필요 |
| Builder | 1 | 1 | 0 | 2 | 1 | **1.0** | 코드 충분 |
| QC&Ranker | 0 | 0 | 0 | 1 | 5 | **1.2** | 코드 충분 |
| DiversityMgr | 0 | 0 | 0 | 1 | 4 | **1.0** | 코드 충분 |
| Critic | 4 | 5 | 3 | 3 | 2 | **3.4** | LLM 필요 |
| Reporter | 4 | 2 | 2 | 2 | 1 | **2.2** | LLM 유익 |

#### 4.2.1 Planner: Qwen 2.5 7B (최적)

Planner가 LLM 능력을 필요로 하는 이유:
- Critic 피드백을 자연어로 해석하여 파라미터 갱신으로 변환
- 과학적으로 일관된 가설 생성
- `FAILURE_ACTION_MAP` 처방을 넘어선 파라미터 공간의 창의적 탐색

Qwen 2.5 7B의 강력한 구조 출력 능력이 Planner의 `ExperimentPlan` 객체 생성 요구와 잘 맞는다.

**프롬프트 전략**: `ExperimentPlan` JSON 스키마 + 가설-파라미터 매핑 퓨샷 예시를 포함한 시스템 프롬프트.

#### 4.2.2 Builder: LLM 불필요 (규칙 기반)

Builder의 `execute_pipeline()`은 전적으로 절차적이다:
1. `plan.steps_config`를 순회
2. 활성화된 각 단계에 대해 `execute_step()` 호출
3. 재시도/폴백 정책 적용 (`handle_failure()`)
4. 결과를 `shared_state`에 기록

이 흐름에서 언어 이해나 창의적 추론이 필요한 결정은 없다. **LLM 호출을 직접 `BuilderAgent.execute()` 호출로 대체하면 능력 손실 없이 LLM 왕복 1회를 절약한다.**

#### 4.2.3 QC&Ranker: LLM 불필요 (순수 알고리즘)

QC&Ranker는 4개의 순차 게이트 평가를 수행한다:
1. **pLDDT 게이트**: `plddt_mean >= 임계값` (수치 비교)
2. **도킹 게이트**: 점수 기준 상위 K% 선택 (정렬 + 슬라이싱)
3. **Rosetta 게이트**: `ddG <= 임계값 AND clash <= 10` (수치 비교)
4. **선택성 게이트**: `margin <= 임계값` (수치 비교)

이후 최소-최대 정규화 및 가중 선형 결합으로 최종 점수를 산출한다. 모든 연산이 표준 수치 알고리즘이다. **LLM은 본질적으로 선형 대수인 작업에 지연과 환각 위험만 추가한다.**

#### 4.2.4 DiversityManager: LLM 불필요 (알고리즘 클러스터링)

DiversityManager가 구현하는 기능:
- 탐욕적 서열 동일성 클러스터링: O(n^2) 쌍별 비교
- FoldMason lDDT 행렬 클러스터링: 사전 계산된 유사성 행렬
- 대표 선택: 클러스터 크기, 이후 `final_score` 기준 탐욕적 선택

해석이 필요한 모호성이 없는 잘 정의된 알고리즘들이다. **LLM 개입은 재현 가능해야 하는 클러스터링 단계에 비결정성을 도입한다.**

#### 4.2.5 Critic: Qwen 2.5 7B + FAILURE_ACTION_MAP 퓨샷 (가장 중요한 역할)

Critic은 가장 인지적으로 요구되는 에이전트로, 다음을 필요로 한다:
- **근본 원인 분석**: 후보가 여러 품질 차원에서 실패한 이유 해석
- **인과 추론**: 실패 패턴을 파라미터 조정으로 연결
- **가설 생성**: 검증 가능한 과학적 가설 수립
- **제약 충족**: 제안을 2개 파라미터로 제한 (추적 가능성 요건)

기존 `FAILURE_ACTION_MAP`(6개 실패 유형 x 1~3개 조치 = 총 13개 매핑)이 LLM을 위한 훌륭한 **퓨샷 발판**을 제공한다.

**위험 완화**: 출력 스키마 검증(JSON: `proposed_changes: [{parameter_name, old_value, new_value, rationale}]`)으로 환각된 파라미터명 포착. 검증 실패 시 규칙 기반 `propose_changes()`로 폴백.

#### 4.2.6 Reporter: Qwen 2.5 7B (템플릿 강화)

Reporter가 LLM 능력의 혜택을 받는 부분:
- 반복 서사의 자연어 종합
- 실험 노트북 항목의 맥락 의존적 강조
- 반복 간 결과 추세에 대한 적응적 해설

그러나 Reporter 출력의 상당 부분은 템플릿 기반이다(PyMOL 스크립트, CSV 열, 마크다운 테이블). **기본적 지시 따르기가 가능한 7~8B 모델이면 충분하다.** 일관성을 위해 Qwen 2.5 7B를 권장하나, 가장 대체 가능한 LLM 역할이다.

## 8. 결론 및 향후 과제

### 8.1 핵심 결론

1. **6개 에이전트 분해가 아키텍처적으로 건전하다.** 관심사 분리가 과학적 워크플로우(계획-실행-평가-비평-보고)와 정렬되며, 강한 테스트 가능성과 확장성 속성을 제공한다. 4개 에이전트로의 통합은 과부하된 MD Agent, 자기 QC 편향, 확대된 장애 영향 범위를 초래하며, 이를 상쇄할 충분한 이점이 없다.

2. **6개 중 3개 에이전트만 LLM 추론을 필요로 한다.** Builder, QC&Ranker, DiversityManager는 LLM 개입이 지연, 비결정성, 환각 위험만 추가하는 순수 알고리즘 작업을 수행한다. 이 인사이트로 모든 에이전트에 LLM을 적용하는 것 대비 LLM 호출을 50% 줄인다.

3. **Qwen 2.5 7B-Instruct가 최적의 단일 모델 선택이다.** 강한 MMLU (74.2), 네이티브 도구 호출, 안정적 JSON 출력, 128K 컨텍스트 윈도우의 조합이 AG_src 에이전트 작업의 모든 관련 차원에서 대안을 능가한다.

4. **LLM 통합이 성공적으로 완료되었다.** Planner, Critic, Reporter 에이전트가 Ollama 기반 Qwen 2.5 7B를 통해 실제 LLM 추론을 수행하며, 실패 시 규칙 기반 폴백이 동작한다. 실제 파이프라인 실행에서 LLM 생성 가설, 파라미터 변경 제안, 보고서가 성공적으로 생성되었다.

5. **Critic 에이전트가 가장 높은 위험의 LLM 통합 지점이다.** 실패 분류와 인과 추론 작업이 7B 모델의 한계에 도전한다. `FAILURE_ACTION_MAP` 퓨샷 발판과 규칙 기반 폴백이 필수 안전망이다.

### 8.3 향후 과제

1. **적응형 LLM 라우팅**: 신뢰도 기반 라우팅 메커니즘 구현. Critic이 먼저 규칙 기반 분류를 시도하고, 신뢰도가 임계값 미만이면 LLM 추론으로 에스컬레이션. 불필요한 LLM 호출을 추가 감소.

2. **다중 모델 앙상블 (Critic)**: Critic 역할에 두 개의 다른 7B 모델(Qwen 2.5 + Mistral 7B)을 실행하고 합의를 취하여 단일 모델 환각 위험 감소.

3. **파이프라인 결과 기반 강화 학습**: Critic이 제안한 파라미터 변경이 후속 반복에서 실제로 ddG를 개선했는지 추적. 이 신호로 시간 경과에 따라 Critic의 퓨샷 예시를 미세 조정.

4. **에이전트 간 메시지 버스 활성화**: `BaseAgent` 메시징 인프라가 완전 구현되었으나 미사용 상태. 활성화하면 비동기 에이전트 통신과 이벤트 기반 파이프라인 실행이 가능해져, 독립 단계의 병렬화를 지원.

5. **GLP-1 유사체 안정성 예측 모듈**: 혈액 내 안정성 6~10일 유지 요건에 대한 예측 모듈 개발. GLP-1 기술 접목.

6. **자동화된 벤치마크 스위트**: 알려진 우수 펩타이드 바인더 표적(예: 공개된 SSTR2 리간드)을 사용한 합성 벤치마크 생성. 개별 구성요소 정확성뿐 아니라 종단간 파이프라인 정확도 평가.
