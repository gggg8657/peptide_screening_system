## V-03: Octreotide D-AA 테스트

### 입력 형식 (코드·예제 기준 실측)

- **`calculate_descriptors.py` / `My_GNN` 파이프라인**: **`SMILES` + `SEQUENCE`(1-letter 문자열)** 를 함께 사용 (`example.csv` 참조).
- **그래프 구축 (`construct_RGCN_bigraph_from_smiles`)**: **SMILES만** 사용 (RDKit `MolFromSmiles`).
- **FASTA 파일 입력 API는 제공되지 않음**(저장소 내 미발견).

### Octreotide 서열 처리 한계 (실험 설계 명시)

- 회의 안의 서열: `D-Phe-Cys-Phe-D-Trp-Lys-Thr-Cys-Thr(ol)`
- 본 검증에서는 레포 내 `pipeline_local/scripts/sequence_to_smiles.py`로 **선형 8잔기 근사**만 수행:
  - 1-letter: **`FCFWKTCT`**
  - **D-AA 위치**: `1:D-Phe`, `4:D-Trp` (옥트레오타이드 유사 패턴)
  - **`Thr(ol)`(C 말단 알코올)** 은 스크립트에 전용 치환 부재 → **말단을 통상 `T`(Thr)로 처리**함. 따라서 결과는 **진약 동등물이 아닌 구조 근사**이며, 워크플로 검증 목적에 한함.

### 생성한 SMILES (요약)

- **변형 A (D-Phe + D-Trp)** 및 **변형 B (L-Phe + D-Trp)** 는 `conda run -n peptools python .../sequence_to_smiles.py` 로 생성.
- RDKit **InChI(스테레오 포함)** 비교 시 **두 문자열은 동일하지 않음**(실측 `same inchi: False`). 예: `/t...33+...` vs `...33-...` 구간 차이.

### pepADMET 쪽 실행 결과

#### 1) 전체 디스크립터 (`calculate_descriptors`)

스크립트: `_workspace/pepadmet_local/run_oct_descriptor_test.py`

두 변형 모두 동일하게 실패:

```text
Error: ...SMILES...: type _vectSt6vectorIiSaIiEE doesn't define __round__ method
```

→ ** toxicity 모델이 기대하는 2133차원 FPN 입력을 재현하지 못해**, 저장소에 포함된 `toxicity_early_stop.pth` 기반 **멀티태스크 toxicity 추론은 본 환경에서 완료 불가**.

#### 2) 부분 디스크립터(PyMolecule + PyProtein만, modlamp 제외)

스크립트: `_workspace/pepadmet_local/run_partial_descriptor_diff.py`

- 부분 특성 수: **2123 keys**
- **D-Phe vs L-Phe 두 SMILES 간 값 차이 나는 키 수: 0**

해석:

- **동일 `SEQUENCE` 문자열 `FCFWKTCT`** 를 쓰는 한 PyProtein 기반 부분은 당연히 동일.
- PyMolecule `GetAllDescriptor()` 결과도 본 쌍에서는 수치가 동일하게 나옴(입체 이성질체지만 많은 저차 분자 지표는 동일할 수 있음).

#### 3) GNN 그래프 원자 피처 (실측으로 스테레오 반영 여부 확인)

스크립트: `_workspace/pepadmet_local/run_graph_atom_diff.py`

```text
atom tensor equal? False
max abs diff: 1.0
nodes 72 72
```

→ **`construct_RGCN_bigraph_from_smiles` 가 생성하는 `ndata['atom']` 텐서는 두 스테레오 SMILES 간 다름**(이진 피처 최대 차이 1.0).

동일 노드·간선 수:

```text
D-Phe+D-Trp graph OK nodes: 72 edges: 150
L-Phe+D-Trp graph OK nodes: 72 edges: 150
```

(`run_oct_graph_test.py`)

## D-AA 처리 가능 여부 결론 (실측 기반)

**결론: 부분 가능 (단, 본 저장소·본 conda 스택에서는 “완전한 추론 파이프라인” 불가)**

| 계층 | D-AA / 스테레오 반영 | 근거(실측) |
|------|---------------------|------------|
| SMILES 입력 | 가능 | InChI 스테레오 정보 상이 |
| RGCN 그래프 원자 피처 | 가능 | `atom tensor equal? False` |
| PyMolecule+PyProtein 일부 특성 | 이번 쌍에서는 차이 없음 | 값 차이 0개 키 |
| modlamp 포함 전체 2133 디스크립터 | **불가(현 환경)** | `_vectSt6vectorIiSaIiEE` / `__round__` 예외 |
| `toxicity_early_stop.pth` 추론 | **불가(현 환경)** | 필수 디스크립터 벡터 미생성 |

요약 한 줄: **SMILES와 GNN 입력 그래프 수준에서는 D-AA(키랄) 차이가 반영되지만, 공개 레포가 의존하는 전체 디스크립터 파이프라인이 깨져 있어 ADMET/toxicity 예측까지는 검증하지 못했다.**

## 다음 단계 권고

1. **운영 우선순위**: modlamp `GlobalDescriptor.calculate_all` 경로에서 발생하는 `__round__` 관련 타입 버그를 **저자 환경(재현 Dockerfile/environment.yml) 확보 또는 modlamp 버전 고정**으로 해결해야 `calculate_descriptors.py` 가 의미 있는 출력을 생성함.
2. **GPU 스택**: H100 환경에서는 **CUDA 11.7 기반 torch 1.13** 대신 **sm_90 지원 최신 PyTorch + 매칭 DGL** 재설치 검토.
3. **`pharmacology_guards.py` 등록**: 본 레포 스택에서는 수치 예측 미생성 → 가드에는 **`DATA_UNAVAILABLE` / `PIPELINE_FAILED`** 등 실행 실패 분기와 함께, **“JCIM 웹·통합 플랫폼 endpoint 수 ≠ GitHub 최소 학습 아티팩트”** 문구를 명시하는 편이 안전.
4. **입력 SSOT**: 옥트레오타이드류는 **`Thr(ol)`·디설파이드·환형**까지 포함해야 함 → HELM 또는 검증된 펩타이드 SMILES 외부 소스와의 정합 필요.
